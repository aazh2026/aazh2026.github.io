Claude Code 上周发布了一个功能，叫 Dynamic Workflows。Anthropic 的工程师 Thariq Shihipar 写了一篇文章解释它，2.5M 阅读量。

这个功能本身不复杂：Claude 可以为自己当前的任务写一个定制的 harness——一个协调多个 subagent 的工作流。生成的 harness 不是通用的，而是针对这个任务的需求动态构建的。

但这个功能的深层含义值得认真拆解。

## 三个具体的失败模式

在 Dynamic Workflows 出现之前，Claude 在处理复杂任务时会出现三种具体的失败模式，Thariq 给它们起了名字：

**Agentic laziness**：面对特别复杂的任务，AI 会停在"差不多完成了"的地方。安全审计应该检查 50 个问题，它只做了 20 个就交差。原因是：在单一上下文窗口里，推进任务和判断完成度是同一个过程，AI 会倾向于用"足够努力"代替"真正完成"。

**Self-preferential bias**：AI 倾向于偏好自己的结果。当被要求验证自己的输出时，这个 bias 尤其明显。在同一个上下文里，验证和生成是连续的，验证者知道自己在验证什么，容易被已经生成的内容带着走。

**Goal drift**：上下文窗口在多次压缩之后，最初的目标会逐渐失真。一些边缘要求、"不要做 X"这类约束会在压缩中丢失。这是因为在单一上下文里，新旧信息争夺同一块空间，优先级判断是隐式的而不是显式的。

这三个问题有一个共同根源：**单一上下文窗口**。

## 真正的解法不是优化，是改变架构

过去的解法是在单一窗口里做更好的 prompt engineering。但这三个问题不是 prompt 能解决的——它们是架构固有的问题。

Dynamic Workflows 的思路是：让 Claude 自己生成一个针对当前任务的专用工作流。这个工作流不是一个通用框架，而是根据任务需求动态生成的。

具体怎么做？以 Fan-out-and-synthesize 模式为例：

把一个大任务拆成很多小步骤，每个 subagent 在独立的上下文窗口里处理自己的部分，最后一个综合 agent 把所有结果合并。每个 agent 都有自己的干净上下文，不互相干扰，不在同一个神经回路里竞争资源。

再举一个例子：Adversarial verification。

每个生成的 agent 配一个专门的验证 agent。两个 agent 从不同角度处理同一个问题——生成 agent 想的是"怎么解决这个问题"，验证 agent 想的是"这个解决方案可能在哪里出错"。最后才输出。

这直接解决了 Self-preferential bias——不是让 AI 验证自己的输出，而是让另一个 agent 来验证。两个 agent 在不同的上下文里，有不同的目标函数。

## "先计划再执行"在架构层面的实现

Thariq 的解法和 Matt Van Horn 的技巧在结构上指向同一个方向：把"规划"从"执行"里分离出来，让它们在各自最有利的条件下运行。

Matt 用的是 prompt 结构：让 AI 先写计划，认可了再执行。他发现"先计划再执行"比直接执行产生更深的结果。

Thariq 用的是架构：让不同的 subagent 分别负责规划、执行、验证。规划 agent 不用等执行完成，验证 agent 不用在同一上下文里被生成内容影响。

两者从不同层次证明了同一件事：当规划和执行在同一个上下文里混合时，两者都会变差。分离它们不是可选项，而是必选项。

## 为什么这个分离这么难

这是因为规划和执行在人类思维里就是分离的，但这个分离在工程系统里实现得很晚。

建筑师先画草图再施工。作家先列提纲再写正文。将军先制定战略再指挥战斗。这些不是繁文缛节，它们之所以存在，是因为分开之后，两者都能在最有利的认知条件下运行——草图阶段不需要考虑施工细节，正文阶段不需要重新思考结构。

AI 系统现在面临同样的问题。只不过以前我们没有能力在工程系统内部实现这个分离，只能依靠 prompt 结构来诱导。

Dynamic Workflows 意味着我们第一次可以在架构层面实现这个分离——不是通过人类的自我纪律，而是通过系统设计本身。

## 一个值得注意的细节

Dynamic Workflows 允许不同的 subagent 使用不同的模型。复杂推理任务用 Opus，简单分拣任务用 Sonnet。

这说明模型能力分层已经成为基础设施层面的设计决策，而不是用户侧的选择。模型不再是一个固定选项，而是一个可以被工作流动态调配的资源。

当这个模式成熟之后，"用哪个模型"这个问题会从用户决策变成系统决策。这个转变的影响范围远超过编程工具本身。
