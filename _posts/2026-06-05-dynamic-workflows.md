上周 Claude Code 发布了一个功能：Dynamic Workflows。Anthropic 工程师 Thariq Shihipar 写了一篇详细的技术文章解释它，2.5M 阅读量。这不是营销文，是实打实的技术描述——值得认真读。

## 它解决的是什么问题

当我们让 AI 执行一个复杂的长任务，AI 会遇到三个具体的失败模式：

**Agentic laziness**：面对特别复杂的任务，AI 会停在"差不多完成了"的地方——比如安全审计应该检查 50 个问题，但它只做了 20 个就交差了。

**Self-preferential bias**：AI 倾向于偏好自己的结果。当被要求验证自己的输出时，这个 bias 尤其明显。

**Goal drift**：上下文窗口在多次压缩之后，最初的目标会逐渐失真。一些边缘要求、"不要做 X"这类约束会在压缩中丢失。

这三个问题有一个共同根源：**单一上下文窗口**。一个大脑在同一个上下文里既要做规划又要做执行，既要生成内容又要判断质量，既要推进任务又要注意边界。

## 动态工作流的解法

Dynamic Workflows 的核心思路是：让 Claude 自己为当前任务编写一个定制的 harness——一个专门协调多个 subagent 的工作流。

这个 harness 不是一个通用的框架，而是针对这个任务的特定需求动态生成的。

举几个具体的使用模式：

**Fan-out-and-synthesize**：把一个大任务拆成很多小步骤，每个 subagent 在独立的上下文窗口里处理自己的部分，最后一个综合 agent 把所有结果合并。这个模式解决的是"上下文窗口太挤"的问题——每个 agent 都有自己的干净上下文，不互相干扰。

**Adversarial verification**：每个生成的 agent 配一个专门的验证 agent，两个 agent 从不同角度处理同一个问题，最后才输出。这直接解决 Self-preferential bias——不是让 AI 验证自己的输出，而是让另一个 agent 来验证。

**Tournament**：N 个 agent 各自用不同方法尝试同一个任务，然后一个评判 agent 对它们的结果做两两比较，直到选出最优解。这比直接让 AI 在一次回答里排出优先级要可靠得多——比较判断比绝对评分更稳定。

**Root-cause investigation**：分别用不同 agent 从日志、文件、数据三个不同角度生成独立假设，然后让每个假设面对一组验证和反驳 agent。这解决的是"单一视角"问题——我们要求 AI 调试时生成多个独立假设，但单一上下文窗口会让 AI 不自觉地收敛到第一个看起来合理的假设。

## 为什么这是一件结构性的事

AI 系统的能力边界，很多来自于架构限制而非智力上限。单一上下文窗口就是一个架构限制——它在概念上简单，但在处理超长任务时天然会积累误差。

过去的解法是在单一窗口里做更好的 prompt engineering。现在的解法是：**改变架构本身**——让多个有独立上下文的 agent 协作。

这意味着我们正在把 AI 系统从"一个聪明的大脑"变成"一个聪明大脑的社会"。

后者有完全不同的能力边界——它受限于 agent 之间的协作质量，而不受限于任何单一 agent 的上下文窗口大小。

这和人类解决问题的模式非常接近：复杂问题之所以被解决，不是因为一个人超级聪明，而是因为有一群人各自负责一部分，然后有人综合结果。

Dynamic Workflows 做的事情本质上是这个社会模式在 AI 系统里的工程实现。

## 值得注意的一个细节

文章里提到，Dynamic Workflows 可以让不同的 subagent 使用不同的模型——复杂推理任务用 Opus，简单分拣任务用 Sonnet。

这说明模型的能力分层已经不是固定的定价策略，而是被纳入了动态路由的基础设施层面。模型不再是一个固定选项，而是一个可以被工作流动态调配的资源。

当这个模式成熟之后，"用哪个模型"这个问题会从用户决策变成系统决策。这对 AI 系统的交互方式有深远影响。
