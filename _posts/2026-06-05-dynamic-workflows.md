## 三个名字，三个真实的问题

Thariq Shihipar 在 X 上发了一篇长文，解释 Claude Code 的 Dynamic Workflows 为什么存在。2.5M 阅读量。这不是营销文，是工程解释。

他描述了 AI 在单一上下文窗口里处理复杂任务时会出现的三种失败模式：

**Agentic laziness**。面对特别复杂的任务，AI 会停在"差不多完成了"的地方。安全审计应该检查 50 个问题，它只做了 20 个。不是因为不努力，是因为在同一个上下文里，推进任务和判断完成度是同一个过程——它没有额外的注意力资源来精确评估边界在哪里。

**Self-preferential bias**。AI 倾向于偏好自己的结果。当被要求验证自己的输出时，这个 bias 尤其明显。这是因为在单一上下文里，验证者知道自己在验证什么，知道生成者给出了什么内容——它不是在独立判断，而是在"确认"而不是"验证"。

**Goal drift**。上下文窗口在多次压缩之后，最初的目标会逐渐失真。一些边缘要求、"不要做 X"这类约束会在压缩中丢失。这是因为在单一上下文里，新旧信息竞争同一块空间，优先级判断是隐式的而不是显式的。

这三个问题有一个共同根源：**单一上下文窗口**。

## 不是优化问题，是架构问题

过去的解法是在单一窗口里做更好的 prompt engineering。

但这三个问题不是 prompt 能解决的——它们是架构固有的。一个大脑在同一个上下文里既要做规划又要做执行，既要生成内容又要判断质量，既要推进任务又要注意边界——这些事情在同一个神经回路里竞争上下文资源，结果就是每件事都做不好。

解决方案不是更聪明的模型，而是改变架构本身。

## Dynamic Workflows 在做什么

Dynamic Workflows 的核心思路：让 Claude 自己为当前任务写一个定制的 harness——一个协调多个 subagent 的工作流。这个 harness 不是通用的，而是针对任务的特定需求动态生成的。

具体来看几个模式：

**Fan-out-and-synthesize**。把一个大任务拆成很多小步骤，每个 subagent 在独立的上下文窗口里处理自己的部分，最后一个综合 agent 把所有结果合并。每个 agent 有自己的干净上下文，不互相干扰，不在同一个神经回路里竞争资源。

这解决的是"上下文窗口太挤"的问题——当一个任务需要同时追踪多个方向，每个方向都需要深入，但单个窗口装不下所有信息时，分到多个窗口是最直接的办法。

**Adversarial verification**。每个生成的 agent 配一个专门的验证 agent。两个 agent 从不同角度处理同一个问题：生成 agent 想的是"怎么解决这个问题"，验证 agent 想的是"这个解决方案可能在哪里出错"。最后才输出。

这直接解决 Self-preferential bias——不是让 AI 验证自己的输出，而是让另一个 agent 来验证。两个 agent 在不同的上下文里，有不同的目标函数。验证 agent 不知道生成 agent 给了什么，所以它不是在确认，而是在独立判断。

**Tournament**。N 个 agent 各自用不同方法尝试同一个任务，然后一个评判 agent 对它们的结果做两两比较，直到选出最优解。这比直接让 AI 在一次回答里排出优先级要可靠得多——比较判断比绝对评分更稳定，因为每一次比较都是独立做出的。

**Root-cause investigation**。分别用不同 agent 从日志、文件、数据三个不同角度生成独立假设，然后让每个假设面对一组验证和反驳 agent。这解决的是"单一视角"问题——当单一上下文窗口要求 AI 收敛到第一个看起来合理的假设时，这个假设往往不是最好的。

## 这在结构上和 Matt Van Horn 的技巧相同

Matt 在 X 上发过"Every Agentic Engineering Hack I Know"。他有一个技巧：做深度工作时，不让 AI 直接写，而是先让它写一个计划，告诉他会怎么处理这些材料、怎么组织最后的文档。只有计划让他认可了，才让 AI 开始执行。

这个技巧和 Thariq 的 Dynamic Workflows 在结构上做的是同一件事：**在规划阶段和执行阶段之间制造分离，让它们在各自最有利的条件下运行**。

Matt 用的是 prompt 结构：改变请求的结构，从"给我 X"变成"先告诉我你会怎么做 X，然后做 X"。Thariq 用的是多 agent 架构：让不同的 subagent 分别负责规划、执行、验证。

两者从不同层次证明了同一件事：当规划和执行在同一个上下文里混合时，两者都会变差。分离它们不是可选项，而是必选项。

## 为什么这个分离这么难——以及为什么现在能做成了

这是因为规划和执行在人类思维里就是分离的，但这个分离在工程系统里一直实现得很晚。

建筑师先画草图再施工。作家先列提纲再写正文。将军先制定战略再指挥战斗。这些不是繁文缛节，它们之所以存在，是因为分开之后两者都能在最有利的认知条件下运行——草图阶段不需要考虑施工细节，正文阶段不需要重新思考结构。

AI 系统现在面临同样的问题。但以前我们没有能力在工程系统内部实现这个分离——因为需要让 AI 能够自主决定把哪些能力分配给哪个 subagent，以及如何协调它们。直到模型能力达到了某个阈值，这个架构才成为可能。

## 真正重要的细节

Dynamic Workflows 可以让不同的 subagent 使用不同的模型。复杂推理任务用 Opus，简单分拣任务用 Sonnet。

这说明模型能力分层已经不是固定的定价策略，而是被纳入了动态路由的基础设施层面。模型不再是一个固定选项，而是一个可以被工作流动态调配的资源。

当这个模式成熟之后，"用哪个模型"这个问题会从用户决策变成系统决策。用户不需要知道 Opus 和 Sonnet 的差别，工作流会自己决定每个子任务分配给哪个模型最合适。

这意味着，对于真正复杂的问题，答案的质量不只是取决于模型有多聪明，而是取决于我们有没有为它建立一个能够让规划和执行各自充分展开的结构。Dynamic Workflows 是第一次在工程层面证明这个结构的有效性。
