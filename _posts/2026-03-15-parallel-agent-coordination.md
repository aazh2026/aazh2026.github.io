## 一个激进的实验

2025年末，Anthropic 研究员 Nicholas Carlini 做了一个大胆的尝试：

让 **16 个 Claude Agent** 并行工作，从零开始构建一个 C 编译器——不是玩具项目，而是能编译 Linux 内核的工业级编译器。

没有项目经理。
没有每日站会。
没有中央协调者。

只有简单的机制：
- 文本文件作为任务锁
- Git 作为同步原语
- 每个 Agent 自主决定"下一个最明显的问题"

7个月后，他们交付了：
- **10 万行 Rust 代码**
- **能编译 Linux 6.9（x86/ARM/RISC-V）**
- **2000 次 Claude Code 会话**
- **$20,000 API 成本**

这个实验不仅产出了一个能工作的编译器，更揭示了一个被忽视的领域：**多 Agent 协调机制**。

---

## 为什么需要并行 Agent？

单 Agent 的限制很明显：

1. **单线程瓶颈**：一个 Claude Code 会话一次只能做一件事
2. **无法并行调试**：大型项目中，多个问题需要同时解决
3. **专业化缺失**：没有专门负责文档、代码质量或子任务的 Agent

Carlini 的解决方案是激进的：**去中心化并行**。

---

## 协调机制的设计

### 极简主义的基础设施

```bash
# 每个 Agent 运行在独立的 Docker 容器中
# 共享一个 bare git repo
/upstream    # 共享仓库
/workspace   # Agent 本地克隆
```

### 任务锁：文本文件 + Git 同步

协调的核心机制简单到近乎荒谬：

```bash
current_tasks/
├── parse_if_statement.txt
├── codegen_function_definition.txt
└── optimize_register_allocation.txt
```

**协议**：
1. Agent 想要处理任务？写一个文件到 `current_tasks/`
2. 如果两个 Agent 争抢同一任务？Git 同步强制第二个 Agent 选择不同任务
3. Agent 完成工作？Pull 上游变更 → Merge 其他 Agent 的改动 → Push → 删除锁文件

**没有中央调度器。没有复杂的状态管理。只有 Git 的原始力量。**

### 合并冲突？让 Agent 自己解决

在并行执行中，合并冲突是常态而非例外。

Carlini 的发现：**Claude 足够聪明，可以自己解决大部分冲突。**

这改变了一个基本假设：
- 传统思维：冲突是人类协调者的责任
- Harness 思维：冲突是 Agent 工作流的一部分

## 自主行为模式

### 无 Orchestrator 的决策

Carlini 明确说明：

**"我没有实现任何其他 Agent 间通信方法，也没有使用编排 Agent。"**

每个 Claude 实例被赋予完全自主权：
- 选择"下一个最明显的问题"
- 维护失败方法的运行文档
- 跟踪剩余任务列表
- 决定何时提交、何时继续、何时放弃

在 git 历史中可以观察到这种行为：Agent 们轮流锁定各种任务，自主协调工作节奏。

### 涌现的协调

令人惊讶的是，这种极简机制产生了有效的协调：

- Agent 们自然地分散到不同任务
- 没有两个 Agent 长期争抢同一任务
- 系统以稳定的速度推进

这不是设计出来的，而是**涌现**出来的。

## 与 OpenAI 方法的对比

Carlini 的去中心化方法与 OpenAI 的 Harness 工程形成有趣对比：

| 维度 | Carlini C编译器 | OpenAI Harness |
|------|----------------|----------------|
| **Agent 数量** | 16 并行 | 3→7 工程师驱动 |
| **协调模式** | 去中心化（无 orchestrator） | 中心化（人类+Agent 混合评审） |
| **任务粒度** | 细粒度（函数级） | 粗粒度（PR 级） |
| **同步机制** | Git + 文件锁 | Ralph Wiggum Loop（自评审循环） |
| **持续时长** | 单任务 6 小时（无人值守） | 平均 3.5 PRs/人/天 |
| **人类介入** | 最小化（仅环境设计） | 中等（PR 评审、任务分配） |

两种方法都有效，但适用于不同场景：
- **Carlini 方法**：适合明确定义的大型工程任务（如编译器），可以无人值守运行
- **OpenAI 方法**：需要更多产品判断的场景（如内部 Beta 产品），人类保留更多控制权

## 去中心化协调的挑战

### 挑战 1：任务依赖管理

简单任务锁无法表达"任务 B 依赖任务 A"的关系。

Carlini 的应对：
- 大部分任务被设计为相对独立
- Agent 遇到依赖时，自己决定是等待还是尝试解决

**局限**：这种方法适合编译器这种模块化程度高的项目，对于高度耦合的系统可能需要更复杂的依赖图。

### 挑战 2：全局一致性

没有中央 orchestrator，如何保证系统整体方向正确？

Carlini 的答案：**高质量的测试套件作为隐性协调机制**。

测试定义了"正确"的标准，所有 Agent 都朝着通过测试的方向努力。这是一种**目标层面的协调**，而非**过程层面的协调**。

### 挑战 3：异常处理

Agent 可能做出破坏性操作。Carlini 观察到的例子：

**"有一次我看到 Claude 意外执行 `pkill -9 bash`，杀死了自己和整个循环。"**

应对：
- 容器化隔离
- 定期人类检查（虽然频率很低）
- 从失败中恢复的机制（重新启动 Agent）

## 工程启示

### 1. 简单性优于完备性

Carlini 的机制极其简单——文本文件锁 + Git。没有分布式事务，没有状态机，没有复杂协议。

但正是这种简单性，让系统**可理解、可调试、可扩展**。

### 2. 信任 Agent 的自主性

传统工程思维倾向于严格控制。Carlini 的实验表明：**当给定正确的目标和反馈机制，Agent 可以自我协调。**

这不是说完全不需要监督，而是说监督的粒度可以更粗。

### 3. 测试即协调

在没有显式协调者的情况下，**测试套件成为隐性的协调机制**。

所有 Agent 都朝着同一个目标优化：通过测试。这比任何人类分配的任务都更清晰、更客观。

## 未来方向

Carlini 的实验只是开始。多 Agent 协调领域还有大量开放问题：

### 研究方向 1：显式依赖图

简单文件锁无法表达复杂依赖。未来的系统可能需要：
- 任务依赖图的形式化表示
- 自动拓扑排序
- 动态依赖发现

### 研究方向 2：Agent 间通信

Carlini 没有实现 Agent 间直接通信。但某些场景可能需要：
- 知识共享（"我发现这个模式有效"）
- 冲突预警（"我要修改你正在用的函数"）
- 协作策略（"我们一起解决这个大任务"）

### 研究方向 3：混合人机协调

完全去中心化 vs 完全中心化是光谱的两端。中间地带可能更有价值：
- Agent 自主处理常规任务
- 人类介入关键决策
- 动态调整自主度

---

## 结语

Carlini 的并行 Agent 实验，展示了一个反直觉的可能性：

**复杂的工程任务，可以用极简的协调机制完成。**

这不是因为 Agent 比人类更聪明，而是因为：
- 清晰的目标（通过测试）替代了复杂的指令
- 简单的同步原语（Git + 文件锁）足以处理大多数协调需求
- Agent 的自主性减少了中央调度的负担

这对软件工程的启示是深远的：

我们长期以来假设大型工程需要复杂的项目管理、详细的任务分配、频繁的同步会议。但也许，**当系统被正确设计，工程可以更像一个自组织的生态系统**。

Harness 工程的真正突破，不是让 Agent 能写代码，而是让**多个 Agent 能协作写代码**。

分布式系统的理论，正在 AI Agent 领域获得新的生命。
