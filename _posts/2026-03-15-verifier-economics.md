## 一个反直觉的发现

2021年，OpenAI 的研究员 Karl Cobbe 等人发表了一篇论文，标题平淡无奇：《Training Verifiers to Solve Math Word Problems》。

但他们的发现一点都不平淡。

在 GSM8K（一个包含 8500 道小学数学应用题的数据集）上，他们比较了两种方法：

| 方法 | 准确率 | 数据扩展性 |
|------|--------|-----------|
| 基础微调（直接生成答案） | ~50% | 线性增长，快速饱和 |
| **训练验证器筛选多个候选答案** | **~70%** | **超线性增长** |

这个差异不仅仅是一个技术优化。它触及了一个更深层的问题：

**在 AI 时代，什么能力最稀缺？**

---

## P vs NP 的直觉

计算复杂性理论中有一个著名的未解决问题：P vs NP。

- **P**：可以在多项式时间内**求解**的问题
- **NP**：可以在多项式时间内**验证**解的问题

核心问题是：求解和验证是否一样难？

虽然这个问题在数学上尚未解决，但在**实际工程**中，答案似乎是明确的：**验证比求解更容易**。

思考一个例子：
- 解一个复杂的数独谜题可能需要数小时
- 但验证一个完成的数独是否正确只需要几分钟

Cobbe 等人的发现，把这个直觉**实证化**了：在语言模型的场景下，训练一个验证器来评判答案，比训练模型直接生成正确答案更有效率、更容易规模化。

## 验证器在 Harness 工程中的映射

OpenAI 和 Anthropic 的两个独立实验，完美印证了这一理论。

### OpenAI 的 Harness 工程

5 个月，100 万行代码，0 手写。

但他们的关键洞察不是"Agent 能写代码"，而是：

**"瓶颈很快从代码生成转移到了人类 QA 容量。"**

解决方案？
- 让应用 UI、日志、指标对 Codex **可读**
- Agent 可以查询 LogQL 和 PromQL 验证自己的行为
- 测试成为验证器，约束 Agent 的输出

### Carlini 的 C 编译器项目

Anthropic 研究员 Nicholas Carlini 的任务更加激进：让 16 个并行 Claude Agent 从零写一个能编译 Linux 内核的 C 编译器。

结果：
- 2000 次 Claude Code 会话
- $20,000 API 成本
- 10 万行 Rust 代码
- **能编译 Linux 6.9（x86/ARM/RISC-V）**

但 Carlini 反复强调：

**"我大部分精力都花在了为 Claude 设计周围的环境——测试、环境、反馈机制。"**

**"测试验证器必须近乎完美，否则 Claude 会解决错误的问题。"**

他遇到的典型问题：
- Claude 频繁地在实现新功能时破坏现有功能
- 解决方案：严格的 CI 管道 + 回归测试

这不是巧合。这是验证器经济学的铁律：

> **没有高质量验证器，生成能力越强，系统越混乱。**

## 验证器设计原则

基于这些经验，我们可以总结 Harness 工程中验证器的设计原则：

### 1. 为 Agent 设计，而非为自己设计

Carlini 的关键洞察："我必须不断提醒自己，我是在为 Claude 写测试，而不是为自己。"

人类可以容忍的测试输出（如冗长的日志），对 Agent 可能是灾难：
- **上下文窗口污染**：数千行无用字节会挤占有用信息
- **信息密度**：错误应该写成 `ERROR: 原因`，方便 grep
- **预计算摘要**：不要让 Agent 重新计算统计信息

### 2. 验证器的验证器

验证器本身也需要被验证。Carlini 的方法：
- 观察 Claude 犯的错误模式
- 设计针对性的新测试
- 持续迭代测试套件

这是一个元级别的反馈回路。

### 3. 从生成到筛选的范式转移

不要试图让 Agent 一次性生成正确答案。相反：
- 让 Agent 生成**多个候选方案**
- 用验证器**筛选**最佳方案
- 如果都不合格，反馈给 Agent 重新生成

这正是 Cobbe 论文中的方法：生成 100 个答案，验证器选出最好的。

## 对程序员职业的影响

验证器经济学对我们意味着什么？

### 技能重心的转移

| 传统技能 | 新技能 |
|---------|--------|
| 写出好代码 | **定义什么叫好代码** |
| 调试程序 | **调试约束系统** |
| 优化性能 | **优化反馈回路** |
| 记忆语法细节 | **设计验证策略** |

### 评判能力 > 实现能力

OpenAI 的实验表明：当 Agent 可以生成 100 万行代码时，限制因素不再是"谁能写得更快"，而是"谁能判断哪些代码是对的"。

这正是 Harness 工程中人类的角色：
- 设计验证器
- 定义"正确"的标准
- 识别输出哪里不对
- 判断方向是否正确

## 实践框架：构建你的验证器系统

如果你想在自己的项目中应用验证器经济学，以下是具体步骤：

### 阶段 1：基础反馈回路

确保 Agent 有最基本的验证能力：
- 可运行的测试套件
- 可解析的 CI 输出
- 指向修复方向的错误信息

这是门槛。Carlini 展示过，即使简单的提示词，配合精心设计的测试基础设施，也能让 Agent 构建复杂系统。

### 阶段 2：领域特定验证

用你系统的特有知识校准验证器：
- 编码架构规则的自定义 linter
- 黄金准则（编码团队审美标准）
- 修复指引文档

OpenAI 的经验：他们曾经每周花 20% 时间清理"AI 垃圾代码"，直到把标准编码进 Harness 本身。

### 阶段 3：元验证

建立验证器的验证机制：
- 测试套套的覆盖率追踪
- 验证器发现问题的统计
- 持续改进的反馈回路

## 更深层的哲学

验证器经济学触及了一个古老的认知真理：

**知道什么是错的，比知道什么是对的更容易。**

在科学中，证伪比证实更容易。在工程中，识别坏设计比发明好设计更直观。在软件开发中，Code Review 往往比从头写代码更快。

Harness 工程只是把这个直觉**系统化**了。

当我们接受"验证比生成更容易"这一前提，整个工程范式就发生了变化：
- 从追求"完美的一次性生成"到"快速迭代 + 严格筛选"
- 从"人类写代码"到"人类设计验证器，Agent 生成候选"
- 从"代码行数作为产出指标"到"验证覆盖率作为质量指标"

---

## 结语

Cobbe 等人的论文标题很谦虚：《Training Verifiers to Solve Math Word Problems》。

但他们的发现指向了一个更大的图景：在 AI 时代，**评判能力**正在取代**实现能力**，成为最稀缺的工程技能。

这不是人类的退却。相反，这是人类角色的升级：

- 从**执行者**到**设计者**
- 从**实现细节**到**定义标准**
- 从**写代码**到**写验证代码的代码**

Harness 工程的实践者们已经证明：当验证器被正确设计，Agent 可以成为可靠的生产力倍增器。

而人类的真正价值，在于知道**什么是好的**——以及如何让机器也明白这一点。
