## 一个失败的开始

OpenAI 的 Harness 工程实验开始时，团队犯了一个错误——这个错误你我可能都犯过。

他们写了一个巨大的 `AGENTS.md` 文件。

想象一下：一份试图涵盖所有规则、所有模式、所有注意事项的百科全书式文档。团队满怀信心地认为，只要把足够的知识塞进去，AI Agent 就能像经验丰富的工程师一样工作。

结果？惨败。

**失败的原因很典型**：

1. **上下文是稀缺资源**。当 Agent 的上下文窗口被一本百科全书占据，留给实际任务、代码和相关文档的空间就被挤压殆尽。Agent 要么错过关键约束，要么开始优化错误的目标。

2. **"什么都重要 = 什么都不重要"**。当文档中的每一条规则都被标记为"关键"，Agent 失去了优先级判断能力，只能在局部进行模式匹配，而不是有意识地导航。

3. **文档迅速腐烂**。一本单片化的手册很快就会变成"规则的墓地"——过时的约束、失效的假设、无人维护的警告混杂在一起。Agent 无法分辨哪些仍然有效，人类也懒得维护，最终这份文档成为系统的拖累。

4. **无法机械化验证**。单个 Blob 文档不适合进行覆盖率检查、新鲜度验证、归属追踪或交叉链接检查。漂移是不可避免的。

这不是 OpenAI 独有的问题。任何尝试过用大型知识库驱动 AI 的团队，都可能遇到同样的困境。

---

## 范式转移：从百科全书到地图

OpenAI 的解决方案简单而深刻：

**把 `AGENTS.md` 当作地图，而不是百科全书。**

具体实践：

```
AGENTS.md (约100行)          → 作为目录/入口点
docs/ (结构化知识库)         → 作为真相来源
```

这不是简单的"拆分文档"，而是一种根本性的认知重构：

### 渐进式披露

Agent 不再被大量信息淹没 upfront。它从一个**小且稳定的入口点**开始，被教导"下一步去哪里看"，而不是一次性加载所有内容。

就像人类阅读技术文档一样：先看目录，再深入感兴趣的章节。

### 机械验证

结构化知识库可以被自动化工具验证：
- 专用 linter 检查文档格式和交叉链接
- CI 作业验证知识库的新鲜度和结构完整性
- 覆盖率工具追踪哪些代码有对应文档

### 文档园丁 Agent

OpenAI 部署了一个专门的 Agent："doc-gardening agent"。

它的工作是：扫描陈旧或与现实代码行为不符的文档，自动提交修复 PR。知识库的维护不再完全依赖人类的主观意愿。

## 知识库的系统架构

OpenAI 的知识库不是随意堆砌的文档集合，而是一个精心设计的系统：

```
docs/
├── beliefs/           # 核心信念与原则（验证状态追踪）
├── architecture/      # 架构文档（领域与包分层地图）
├── quality/          # 质量评估（各领域的健康度追踪）
├── plans/            # 执行计划（进行中与已完成）
└── debt/             # 已知技术债务
```

### 设计文档的目录化

每份设计文档都有元数据：
- **验证状态**：这份文档是否仍然反映现实？
- **核心信念**：驱动这份设计的根本假设是什么？
- **所有者**：谁负责维护？

### 计划作为一等公民

在 Harness 工程中，计划不是临时的待办事项，而是**版本化的产物**：
- 小型变更：轻量级 ephemeral 计划
- 复杂工作：详细的执行计划（execution plans），包含进度日志和决策记录

所有计划都入库，Agent 可以在不依赖外部上下文的情况下操作。

## 为什么这对人类也很重要

Harness 工程的知识管理实践，本质上是在解决一个古老问题：**如何让新成员快速理解系统？**

传统入职流程：
- 阅读几百页文档
- 参加若干次会议
- 花几个月"耳濡目染"

Harness 工程方法：
- 从 100 行的 `AGENTS.md` 开始
- 按需深入结构化的知识库
- 所有知识都是可执行、可验证的

**讽刺的是**：我们为人类设计的知识管理系统，往往不如为 Agent 设计的有效。

## 实践指南：设计你的知识地图

如果你正在或计划引入 AI Agent 到你的工程流程，以下是基于 OpenAI 经验的具体建议：

### 1. 限制 `AGENTS.md` 的长度

目标：100 行以内。它应该回答三个问题：
- 这个系统的核心目标是什么？
- 从哪里开始理解架构？
- 如何找到特定领域的详细文档？

### 2. 建立机械验证机制

为知识库设计 CI 检查：
- 所有文档文件必须有最后更新日期
- 交叉链接必须有效
- 每个模块必须有对应的架构说明

### 3. 任命"文档园丁"

无论是人类还是 Agent，必须有人/系统负责：
- 定期扫描陈旧文档
- 对比代码行为与文档描述
- 主动发起修复

### 4. 渐进式采纳

不要试图一次性重构所有文档。从新建模块开始：
- 新模块使用 Harness 知识管理模式
- 旧模块逐步迁移
- 允许两者共存，直到旧模块被重构或淘汰

## 更深层的含义

Harness 工程的知识管理革命，揭示了一个被长期忽视的真相：

**大多数技术文档之所以无效，不是因为写得太少，而是因为写得太乱。**

人类工程师可以凭借经验和直觉，在混乱的文档中导航。但 Agent 不行——它们需要清晰的结构、明确的指针、可验证的约束。

当我们为 Agent 设计知识管理系统时，我们实际上是在**强迫人类团队澄清自己的思维**。

这不是妥协，而是进化。

---

## 结语

从百科全书到地图的转变，听起来像是一个技术细节。但它实际上代表了软件工程中知识管理的范式转移：

- 从**静态文档**到**活的知识系统**
- 从**人类可读**到**人机双可读**
- 从**被动维护**到**主动验证**

OpenAI 用 5 个月和 100 万行代码证明：当知识被正确结构化，Agent 可以成为真正的生产力倍增器。

而人类工程师的角色，也从"知识的管理者"转变为"知识系统的设计师"。

这正是 Harness 工程的核心洞察。
