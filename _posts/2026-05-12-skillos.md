## 一、一个反直觉的观察

大多数团队在做 Agent 开发时，遇到的第一个问题不是"Agent 做不到什么"，而是"Agent 忘得太快"。

今天帮你完成了一个任务，明天遇到一个几乎一样的任务，它需要你重新描述所有背景。后天遇到第三个变体，你又开始从零解释。

**这不像人类。** 人类会从重复中提取模式，把经验变成直觉。

问题出在架构上。大多数 Agent 框架的默认思路是：**你（人类）告诉 Agent 什么时候该用技能**。你写 prompt、写规则、写技能描述，然后期待 Agent 在正确的时候调用它们。

这有两个根本性缺陷：

1. **你不可能预判所有需要技能的时机** — 现实任务的多样性永远超出你的规则覆盖范围
2. **技能的使用时机本身就是一种知识** — 这种知识不应该由规则传递，而应该从经验中涌现

SkillOS 提出的核心问题：**能不能让 Agent 自己学会什么时候该积累技能？**

## 二、SkillOS 的核心设计

### 2.1 Executor / Curator 分离

SkillOS 把 Agent 的行为分成两个角色：

- **Executor**：执行具体的任务。比如"帮我查一下竞品的定价"。它不关心技能的积累，只管把眼前的事做好。

- **Curator**：在后台观察 Executor 的行为，判断哪些经验值得被固化成可复用的技能。它的工作不是执行任务，而是**识别模式**。

这个分离的洞察在于：**执行和判断是两种不同的认知模式，放在同一个 Agent 里互相干扰**。

Executor 专注于眼前任务时，不应该分心去思考"这个经验以后还能用吗"。Curator 在做判断时，不需要跟踪每一个具体的执行步骤。

### 2.2 三个核心组件

SkillOS 的架构围绕三个组件展开：

1. **Skill Registry**：技能的存储和索引系统。每个技能有明确的输入输出规范，以及适用场景的描述。

2. **Curator Agent**：负责监控执行过程，判断哪些行为值得被记录为技能。判断标准包括：频次（这个模式出现了多少次）、效果（固化后能节省多少 token）、独立性（这个技能能不能脱离当前上下文独立工作）。

3. **Executor Agent**：使用 Curator 积累的技能来执行任务。它不需要知道技能是怎么来的，只需要能用。

## 三、Grouped Task Streams

### 3.1 单独任务的问题

如果你让 Agent 完成一个"查竞品定价"的任务，然后问："这个任务的经验能复用吗？"

答案是：**不知道**。

单次任务暴露不了足够的模式。你无法判断这是不是一个重复性场景，还是一次性例外。

**Curator 需要看到任务之间的关系，而不是任务本身。**

### 3.2 Grouped Task Streams 的设计

SkillOS 引入了"任务流"的概念：不是记录单个任务，而是把相关的任务归类成一个 stream，持续追踪。

比如一个处理用户退款的 stream：

```
[Task 1] 用户申请退货 → 检查库存 → 通过退货
[Task 2] 用户申请换货 → 检查库存 → 换货
[Task 3] 用户申请退货 → 检查库存 → 库存不足 → 等待补货
[Task 4] 用户申请退货 → 检查库存 → 通过退货
```

当 Curator 看到这个 stream，它能发现：

- **"检查库存 → 决定是否通过退货"** 是一个高频模式
- **库存不足的情况** 需要特殊处理分支
- **退货和换货的流程** 共享同一个前置检查

这些模式，单个任务看不到，放到任务流里才能浮现。

## 四、RL for Skill Curation

### 4.1 什么时候该固化一个技能？

这是 SkillOS 最有意思的部分：Curator 不是靠规则判断该不该积累技能，而是用 RL 来学习判断。

固化和不固化，都有成本：

- **固化的成本**：写技能规范、调试、维护
- **不固化的成本**：下次遇到类似任务需要重新描述

Curator 的 RL 目标函数是最小化两个成本的和：**长期token消耗 + 技能维护成本**。

### 4.2 反馈从哪里来？

RL 训练需要反馈。SkillOS 的反馈来自三个方面：

1. **执行效果**：任务完成后，curator 收到任务是否成功的信息
2. **Token 消耗**：如果固化后的技能被复用了，节省了多少 token
3. **用户反馈**：某些实现里，用户会对"这个技能好不好用"打分

这些反馈构成了 curator 的奖励信号，让它学会在正确的时候做固化决策。

## 五、Meta-Skills 的涌现

### 5.1 什么是 meta-skills？

当 curator 积累了足够多的技能，它开始发现**技能之间的关系**。

比如 curator 发现：
- "检查库存"和"更新库存"总是配对出现
- "处理退货"和"生成退款"之间有固定的时序依赖
- 某些异常情况需要人工介入，而另一些可以自动处理

这些**关于技能的知识**，本身就是一种更高层次的技能——meta-skill。

### 5.2 meta-skill 的价值

Meta-skills 让 Agent 能在更抽象的层次做推理：

- 不是"这个技能能不能用"，而是"这个技能和其他技能的组合关系是什么"
- 不是"这个任务做完了没有"，而是"这个任务在更大的流程里扮演什么角色"

这和人类的"专家直觉"非常像：专家不是记住了所有情况，而是抓住了更深层的结构。

## 六、结论

### 核心收获

SkillOS 的贡献不是发明了一种新的 prompt 技巧，而是一个**让 Agent 自己学会进化的框架**。

几个关键点：

1. **Executor/Curator 分离**解决了执行和判断之间的认知干扰
2. **Grouped task streams**让 curator 能从足够多的相关性中提取模式
3. **RL 训练 curator**让技能积累的决策本身也可以学习和优化
4. **Meta-skills 的涌现**展示了从经验中提取更高层次知识的可能性

### Paul Graham 视角

这让我想到一个更大的图景：

我们通常把 AI 的进步理解为"模型变强"。但 SkillOS 说的是：**即使模型不变，只要改变积累和管理技能的方式，Agent 的能力也能持续提升**。

这和 Paul Graham 说的"工作即写作，写作即思考"是同一类洞见的不同版本。你以为你在记录经验，实际上你在建立一个能产生新经验的系统。

技能积累不是记忆的延伸，而是认知复利的基础。
