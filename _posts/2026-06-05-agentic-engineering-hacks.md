Matt Van Horn 上周在 X 上发了一篇文章，标题是"Every Agentic Engineering Hack I Know"。613K 阅读量，配套的开源项目加起来 27K+ stars。其中一个叫 last30days的工具，GitHub 上 27,000 个 stars。

他上一次"真正发布过有人用的软件"是什么时候？高中毕业以后。

这个人现在每天开着四到六个 AI session 并行工作，用的是 Claude Max（$200/月）和 Codex Max（$200/月）。这不是在炫耀工具。这是一种新的工作状态。

## 一个反直觉的工作流

传统的 AI 使用方式是对话式的：你问一个问题，AI 给一个回答，你觉得不够好就继续追问。

Matt 的方式完全不同。他的桌面上同时开着：

- 一个 session 在写 plan.md
- 一个 session 在用另一个已经写好的 plan 构建功能
- 一个 session 在跑 last30days，统计他过去一个月的产出
- 一个 session 在修复测试中发现的 bug

他几乎不读 plan.md。他的原话是："Plans are for agents, you silly human."

这句话听起来像是调侃，但背后有一个真实的认知转变：**当你不再需要自己读计划的时候，计划的存在意义就变了。**

计划不再是你理解任务的方式，而是你控制 agent 行为的方式。你用写计划这个动作来确保 AI 不会偷懒、不会停在半成品、不会用最省力的方式交差。计划是一个约束装置，不是一个信息传递装置。

## "Make a plan for the plan" 为什么有效

Matt 最反直觉的一个技巧是：当他要做真正深度的工作时，他不让 AI 直接写文档或做分析，而是先让 AI 写一个"怎么读这些材料、怎么生产这个文档的计划"。

他举了一个具体例子。他约见了 Michael Margolis——GV 前研究合伙人，以 bullseye customer method 著称——讨论一个商业问题。Margolis 让他读一本书，免费的 PDF 在网上。以前的 Matt 会扫一眼然后继续。这次他打开 Claude Code 说："/ce-plan make a plan for the plan。我要给你两样东西：Margolis 的书（PDF），和两小时的 Granola 会议记录（和 Margolis 讨论的完整上下文）。我要你告诉我，你会怎么读这本书、怎么处理记录，然后把这两个东西整合成一份我真正能用的文档。不要现在写那个文档，现在只写计划。"

Claude 花了 45 分钟写了一个 EPIC plan。然后执行那个计划，产出了完整文档。

这背后的机制是什么？

当 AI 被要求直接给出一个深度答案时，它倾向于用最通用的方式组织内容——因为那样最省 tokens，上下文窗口最干净，风险最低。

当 AI 被要求先计划怎么做，它必须在内心先模拟一个"真正好的答案长什么样"。这个模拟过程本身就是深度推理，而不仅仅是执行。换句话说：**要求 AI 先计划，相当于强迫它先想清楚"什么才是一个真正好的答案"，然后才动手。**

这个技巧不需要任何特殊的 prompt engineering。它只需要你改变请求的结构——从"给我 X"变成"先告诉我你会怎么做 X，然后做 X"。

## YOLO 模式的逻辑

Matt 的配置文件里有一行：

```
skipDangerousModePermissionPrompt: true
```

他开了六个并发 session。如果每个 session 都要等他点确认，他就是在当人肉哨兵。六个 AI 同时工作，每一个都在等他的批准——这比不用 AI 还累。

所以他选择了 YOLO 模式：默认允许所有操作，出问题了 git revert。他的安全网是 GitHub，不是逐个确认对话框。

这个选择有一个隐含的逻辑：**当你有可靠的版本控制能力，逐个确认的边际成本超过了它的边际价值。** 安全和效率不是对立的——你需要先有足够的安全网，才能选择效率。

## 双引擎架构

Matt 同时在用两个 $200/月的订阅：Claude Max 和 Codex Max。Claude 负责规划、研究、taste 类的判断；Codex 负责构建、实现、快速的代码生成。

他用 /ce-work --codex 这个命令桥接两个引擎——从 Claude 的 planning session 里直接调起 Codex 的构建能力，不需要切换窗口，不需要复制粘贴。

这个架构有意思的地方在于：它说明"哪个模型"已经不再是一个用户的决策。Matt 本人在选择用什么模型做什么事上花的精力微乎其微——大部分由工作流框架内部处理。

## 一个被忽视的数据点

Matt 过去一年用这套工作流发布了三个产品：

- last30days：GitHub 27,000 stars
- Printing Press：4,000+ stars
- Agent Cookie：刚发布，已成为 Python、Go、GStack、Paperclip 等顶级开源项目的贡献者

这三个产品，任何一个对于一个独立开发者来说都是值得注意的产出。而他是一个人用业余时间做的。

他上一次认真发布软件是高中毕业。他把这叫做"我重新变成了一个能交付产品的开发者"。

这句话值得停下来想想。不是"我学会了用 AI"，而是"我重新变成了开发者"。这意味着在 AI 时代之前，他以为自己失去了这个能力——不是因为不努力，而是因为软件开发的协作复杂度超出了个体能够独立承载的范围。

AI 编程工具的实质不是让 AI 替代开发者，而是把那些超出个体承载能力的协作复杂度——规划、调试、多任务切换——变得可以委托。当这些被委托出去，剩下的才是真正的个体能力：判断什么值得做，判断什么是好的结果。

这是生产力工具变化的一个规律。每当一种工具把过程中的某个瓶颈变成可委托的，原来需要很多人协作才能完成的工作，一个人就能完成。纺织机是这样，编译器是这样，Git 是这样，AI 编程工具也是这样。
