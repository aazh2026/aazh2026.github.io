> 4. **15% 的 capability 是"暗的"** — 技能存在但 agent 不知道什么时候该用它；skillify 让暗的变成亮的

## 一、框架的问题

LangChain 融了 1.6 亿美元。三年开发。 billion-dollar 估值。LangSmith 是真正精密的测试平台：trajectory evals、trace-to-dataset 管道、LLM-as-judge、regression suites、tool 的单元测试框架。他们有所有的组件。

但组件不等于实践。

LangChain 给你的是测试**工具**，从来不告诉你测什么、按什么顺序测、或什么时候算完成。

没有一个 opinionated workflow 说：

1. 这个失败发生了
2. 现在写一个 skill
3. 现在写确定性代码
4. 现在写单元测试
5. 现在写 LLM evals
6. 现在添加 resolver trigger
7. 现在 eval resolver
8. 现在审计重复
9. 现在做 smoke test
10. 现在正确归档

这个循环不存在。你必须从零散的原始组件里自己发明它。

大多数 AI agent 用户仍然根本不测试他们的 agents，因为——他们用的框架给了他们健身房会员卡，没给他们训练计划。

## 二、Skillify 的核心原则

### Latent vs Deterministic

在 thin harness、fat skills 框架里，有一个关键区分：

- **Latent work**：需要判断的工作。同样的输入，不同的输出。需要模型。
- **Deterministic work**：需要精确的工作。同样的输入，同样的输出，每次都一样。不需要模型。

日历 grep 是确定性的。同样的查询，同样的结果，每次都是。不需要模型。

但 agent 却用了 latent 空间去做这件事——启动推理、调用 API、解释结果——而一个三行脚本就能立刻返回答案。

**这就是 bug。不是答案错了，是用错了边。**

### Skillify 是什么

Skillify 是一个动词。当你感到"这不对，agent 不应该这样处理这个"时，你说一句"skillify it"，整个 debugging session 就变成永久基础设施。

不是"这次修好了"，而是"这个错误永远不会再发生"。

## 三、失败案例 1：已经存在于数据库的行程

Garry Tan 问了一个关于十年前商务旅行的简单问题。应该一秒完成。

Agent 实际做了：

1. 调用 live calendar API → blocked（太久了）
2. 尝试邮件搜索 → 结果嘈杂，没有结论
3. 再次用不同参数尝试 calendar API → 还是 blocked
4. 五分钟后，搜索本地知识库 → 立刻找到了

答案一直就在他自己的数据里。3146 个日历文件，跨越 2013 到 2026。已经索引了。已经本地化了。一个 grep 就够了。

Agent 只是没有先去那里查。

### 修复：calendar-recall

```
Skill: calendar-recall
Trigger: [any question about past calendar events]
Deterministic code: grep local calendar files
Test: 10 historical calendar queries
```

修复后，每次被问到历史行程，agent 先查本地知识库，不再走 API。

## 四、失败案例 2：Agentic RAG 的属性错误

RAG retrieval 找到了正确的文档，但 agent 提取了错误的属性。

"找出 YC 所有合伙人并返回他们的名字"——返回的是"YC"而不是合伙人名字。

Agent 在做提取时注意力分散，提取的是最显眼而不是最相关的字段。

### 修复：structured-extraction

```
Skill: structured-extraction  
Trigger: [any extraction task]
Policy: 
  1. First identify the target field name
  2. Then locate that field in retrieved docs
  3. Verify field exists before returning
Tests: [15 edge case extraction scenarios]
Eval: LLM-as-judge on extraction accuracy
```

## 五、Skillify 10 步清单

当一个失败发生，按这个顺序做：

```
Step 1: SKILL.md
        → Write the skill trigger condition
        
Step 2: Deterministic code  
        → Write the precise, model-free version
        
Step 3: Unit tests
        → Test the deterministic code in isolation
        
Step 4: LLM evals
        → How does the LLM handle this case?
        
Step 5: Resolver trigger
        → What condition activates this skill?
        
Step 6: Resolver eval
        → Does the trigger correctly fire?
        
Step 7: DRY audit
        → Is this skill duplicating existing logic?
        
Step 8: E2E smoke test
        → Full stack integration test
        
Step 9: Brain filing
        → Update CLAUDE.md / system prompt
        
Step 10:归档
        → Link skill to failure ticket
```

## 六、延伸思考

Skillify 最有价值的部分不是 10 步清单本身，而是它背后的逻辑：**错误应该产生资产，不应该只是被修复**。

传统软件开发里，修一个 bug 就是修一个 bug。修完，这个 bug 的"知识"就消散了——除非你写进了某个 code review 记录里，否则下次同类问题还会以不同形式出现。

Skillify 把修 bug 变成了建设基础设施的过程。每次失败 → skill + 测试 + 评估。这个 skill 是资产的积累，不是债务的偿还。

另一个关键洞察：**15% 的 capability 是"暗的"**。这来自 check-resolvable 工具的发现——它能识别出系统里存在但没被路由到的技能。

这意味着大多数 AI 系统的能力比它们实际表现出来的更强。问题不是"AI 做不到"，而是"AI 不知道自己能做到"。Skillify 是让暗的变亮的机制。
