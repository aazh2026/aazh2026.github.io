# Software Forgets: Agent Traces 作为记忆的悖论

代码会忘记。这是软件工程的基本事实。原始作者通常不记得当初为什么这样设计——有时候原始作者就是我自己。提交信息是决策的压缩版本，PR 描述记录的是结论而不是推理过程，剩下的都在某个人的脑子里，直到那个人离开公司。

我们试过的工具都不管用。注释需要维护，文档需要更新，wiki 没有人看，decision logs 写完就没人再打开。维护这些"粘合工作"总是输给功能开发。

Hugging Face 最近在一篇短文里提出了一个新视角：agent 其实已经在生产这些 context——搜索、错误、方向转变，都记录在 JSONL 文件里。问题只是这些东西没有归宿，散落在某人的笔记本电脑上。

这个观察是对的，但它揭示的问题比表面看起来更深。

## 记忆的层级

如果我们认真对待"软件记忆"这个问题，会发现它有多个层级，每一层都比上一层更诚实，但也更贵：

**代码本身**——最薄。只记录结果，不记录为什么。你看一段代码，可以知道它在做什么，但不知道为什么要这样做。

**git 历史**——略好。记录了变更轨迹，但高度压缩。一个 commit 通常只有一行文字，写不下多少上下文。

**PR 描述和 commit message**——更浓缩。通常只有结论："性能优化"、"修复 bug"、"重构 XXX"。决策过程消失在几十个字节里。

**文档**——通常过时。写文档的人知道它很快会变得不准确，所以从一开始就动力不足。

**agent traces**——最完整。记录了完整的推理路径：agent 搜索了哪些文件、尝试了哪些方法、遇到了什么错误、为什么改变方向。这是迄今为止最诚实的决策记录。

问题在于：每一层记忆的维护成本都比上一层高。commit message 基本不需要额外成本；文档需要人写、人更新；而 traces，每一秒钟都在自动生产。

## 读者的变化

Hugging Face 提到了一个有趣的趋势：on-call 调查故障的人，越来越可能是 agent。规划重构的也是 agent。agent 可以秒读一万行 context，但前提是有人保持了那些行。

这意味着记忆的读者正在改变。以前，代码的 context 是给下一个工程师看的。现在，它越来越多地是给下一个 agent 看的。

这个转变产生了一个新的不匹配：人类写 context 是为了人类理解——用自然语言，结构松散，随意省略。agent 读 context 的方式不同——它需要的是完整的、明确的、不依赖隐含知识的输入。

我们还没有真正适应这个变化。大多数团队还在用人类友好的方式生产 context，却指望 agent 能够正确理解它们。

## 存储与判断的悖论

Hugging Face 给出的解决方案是把 agent traces 统一存进 Hugging Face Buckets，每一个 session 有持久化的归宿，agent 可以从过去的 session 提取记忆。这是一个工程上合理的方向。

但它解决的是存储问题，不是检索问题。

traces 里记录的信息量是巨大的。一个复杂的 agent session 可能包含数千个决策节点，其中每一个都有搜索、推理、错误、修正的完整路径。如果你要从中找到真正重要的决策——那个最终决定系统架构的决策——你需要一种检索方式，但没有人知道该怎么做。

更根本的悖论在这里：**我们正在用 AI 解决信息丢失问题，但 AI 本身也是信息丢失的原因之一**。

当人类工程师写代码，决策过程至少有部分发生在对话和文档中——这些是显性的、可追溯的。当 agent 写代码，决策过程发生在 token 序列里，被记录为 traces，但如果没有人主动查看和使用这些 traces，它们就只是躺在存储里的字节。

这和 Goodhart 定律有关：一旦某件事被当作衡量指标，它就不再是纯粹的记录工具。agent traces 的存在本身改变了开发者的工作方式——你会开始为了"留下更好的 traces"而工作，而不是为了做出更好的决策。

## 真正的解决方向

Hugging Face 的 buckets 是一个好的开始，但它解决的是表层问题。

更深的问题有两个：

第一，**我们不知道什么值得记住**。一个架构决策和一个临时性 workaround，traces 记录的形式可能是一样的。如果没有任何方式来区分它们的重要性，存储就只是成本。

第二，**记忆的前提是有人记得回头看**。traces 再完整，如果下一个工程师或 agent 从不看它，它就不存在。人类有动机回顾历史（有好奇心、有责任感）；agent 没有——它只处理当前任务。

也许真正的解决方案不是更好的存储，而是**更小的决策空间**——让每次 agent 调用解决的问题足够小、足够清晰，不需要追溯太多历史就能做出好判断。这不是工程问题，这是设计问题。

Fred Brooks 多年前区分的本质复杂度和偶发复杂度在这里依然适用。存储问题是偶发的；判断什么值得记住、什么时候该回头看，这些才是本质问题。
