# Software Forgets: Agent Traces 作为记忆的悖论

代码会忘记。这是软件工程的基本事实。

原始作者通常不记得当初为什么这样设计——有时候原始作者就是我自己。提交信息是决策的压缩版本，PR 描述记录的是结论而不是推理过程，剩下的都在某个人的脑子里，直到那个人离开公司，或者单纯地忘记。

我们试过的工具都不管用。注释需要维护，文档需要更新，wiki 没有人看，decision logs 写完就没人再打开。维护这些"粘合工作"总是输给功能开发。这是 Herbert Simon 几十年前就描述过的：注意力是最稀缺的资源，而维护 context 是消耗注意力但不被计入产出的工作。

Hugging Face 最近在一篇短文里提出了一个新视角：agent 其实已经在生产这些 context——搜索、错误、方向转变，都记录在 JSONL 文件里。问题只是这些东西没有归宿，散落在某人的笔记本电脑上。

这个观察是对的。但它揭示的问题比表面看起来更深。

## 记忆的层级

如果我们认真对待"软件记忆"这个问题，会发现它有多个层级，每一层都比上一层更诚实，但也更贵：

**代码本身**——最薄。只记录结果，不记录为什么。你看一段代码，可以知道它在做什么，但不知道为什么要这样做，或者为什么非这样做不可。

**git 历史**——略好。记录了变更轨迹，但高度压缩。一个 commit 通常只有一行文字，写不下多少上下文。"性能优化"——优化了什么？为什么在这个时候优化？当时的约束条件是什么？没有记录。

**PR 描述和 commit message**——更浓缩。通常只有结论："修复登录 bug"、"重构权限模块"。决策过程消失在几十个字节里。

**文档**——通常过时。写文档的人知道它很快会变得不准确，所以从一开始就动力不足。这不是人的问题，这是激励结构的问题：文档的维护成本由未来的自己承担，而收益是模糊的。

**agent traces**——最完整。记录了完整的推理路径：agent 搜索了哪些文件、尝试了哪些方法、遇到了什么错误、为什么改变方向。这是迄今为止最诚实的决策记录。

问题在于：每一层记忆的维护成本都比上一层高。commit message 基本不需要额外成本——你已经在写代码，只需要多打一行字；文档需要人写、人更新；而 traces，每一秒钟都在自动生产，成本只有存储。

## 读者的变化

Hugging Face 提到了一个有趣的趋势：on-call 调查故障的人，越来越可能是 agent。规划重构的也是 agent。

这意味着记忆的读者正在改变。

以前，代码的 context 是给下一个工程师看的。工程师有好奇心，有责任感，有"我应该理解这个系统"的内驱力。人类读取 context 的方式是探索式的：先看一个大概，知道有问题再去细查某一部分。

agent 读取 context 的方式不同。它需要的是完整的、明确的、不依赖隐含知识的输入。它不会问"这个模块为什么这样设计"，因为它没有被训练出这个直觉；它只会利用你给它的所有信息去执行任务。

我们还没有真正适应这个变化。大多数团队还在用人类友好的方式生产 context，却指望 agent 能够正确理解它们。这就像用中文写了一份操作手册，放在只有英文阅读器的人面前——信息在，技术在，但理解不在。

## 获取教训的问题

traces 之所以值得关注，一个核心原因是一个组织真正重要的知识资产：获取教训（lessons learned）。

软件工程文献里有大量关于这个话题的研究。Phil水平轴上的各种研究都指向同一个结论：软件团队最容易犯的错误，是重复同样的错误。知道某个方法在某个上下文中失败过的人，下一个团队往往不知道这件事。

传统的解决方案是 post-mortems、retrospectives、decision logs。但它们都依赖于人的主动性——有人愿意写，有人愿意读，有人记得在正确的时间查阅。而 agent traces 的出现，理论上可以让这个过程自动化：不需要人的主动性，决策过程已经被完整记录。

问题在于：记录不等于检索。traces 里记录的信息量是巨大的，一个复杂的 agent session 可能包含数千个决策节点。如果你要从中找到真正重要的决策——那个最终决定系统架构的决策，或者那个避免了严重问题的决策——你需要一种检索方式，但没有人知道该怎么做。

## Brooks 的框架在这里也适用

Fred Brooks 在《人月神话》里区分的本质复杂度和偶发复杂度，在这里依然适用。

存储问题是偶发的。Hugging Face 的 buckets 解决的是这个问题——给你一个地方放 traces，让它们不丢失。这是一个工程问题，有工程解法。

但判断力问题是本质的：什么值得记住？什么时候该回头看？traces 里的哪一部分包含了真正重要的经验教训？这些问题没有工具可以替代人来回答。

traces 不会自动变成有用的组织记忆，除非有人定义什么是有用的。这个定义本身，需要的是对你所在领域的深度理解，而不是对工具的熟练掌握。

## 一个更直接的教训

traces 里最值得记录的不是成功，是失败。

成功的决策人人喜欢记录，但 agent 不会从成功中学到比从失败中更多的东西。失败的路径——那个尝试了但发现不对的方向，那次遇到了错误然后转向了另一个方案——才是真正稀缺的知识。

因为它记录了"这个方向不对"，而这个信息除了 traces 之外很难系统性地被捕获。

但这需要人主动去标记。agent 自己不会说"这次失败值得记住"。它只会记录"这个方法在这种情况下返回了错误"。

所以真正的解决方案可能不是更好的存储，而是**更小的决策空间**——让每次 agent 调用解决的问题足够小、足够清晰，不需要追溯太多历史就能判断当前决策的质量。

这是设计和工程问题，不是工具问题。
