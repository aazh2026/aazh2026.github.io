# Software Forgets: Agent Traces 作为记忆的悖论

Hugging Face 最近写了一篇短文，题目是"Software Forgets: Agent Traces Are the Memory"。核心观点很简单：代码有 git，文档有 Notion，沟通有 Slack，但 agent 的执行痕迹——那些搜索、错误、方向转变——只躺在某个人的笔记本电脑里，没有归宿。

这个观察本身很有意思，但它揭示的问题比表面看起来更深。

## 信息的压缩丢失

任何写过代码的人都知道：提交信息是决策的压缩版本。原始作者记得的"为什么这样做"，通常不会完整进入 commit message。PR 描述也好不到哪去——它记录的是结论，不是推理过程。

Paul Graham 在某篇文章里写过：写作就是思考。你以为自己在写代码，但实际上你在写的是对问题的理解。代码只是这个理解的输出。

问题在于：当 agent 开始写代码，**理解的生产者变了**。人类工程师的决策过程至少有迹可循（搜索、错误、方向调整），但这些过程从来没有被正式记录。agent 的执行轨迹是迄今最完整的决策记录——比任何 commit message 都更完整——但它们毫无组织地散落在各处。

这产生了一个悖论：**我们有了更好的记录工具，却更不知道该把它们放在哪里**。

## 谁需要这些记忆？

Hugging Face 提到了一个有趣的变化：on-call 调查故障的人，越来越可能是 agent。重构的规划者，也越来越可能是 agent。

这意味着记忆的"读者"正在改变。以前是给下一个工程师留 context，现在是给下一个 agent 留 context。而 agent 的阅读能力和人类不同——它可以秒读 1 万行 context，但前提是有人保持了那些行。

Hugging Face 的解决方案是把 agent traces 存进 Hugging Face Buckets，让每个 session 都有持久化的地方可存，并且 agent 可以从过去的 session 中提取记忆。这是一个工程上合理的方向。

但我更感兴趣的是这个变化带来的另一个问题：**当 agent 产生的 context 比人类产生的更多、更详细，人类的判断力反而更稀缺了**。因为最终决定一个系统该怎么构建的，还是人。agent 可以记录所有路径，但选择走哪条路的还是人。

## 记忆的层级

如果我们认真思考"软件记忆"这个问题，会发现它有多个层级：

1. **代码本身**：最薄，只记录结果
2. **git 历史**：略好，记录了变更轨迹，但压缩严重
3. **PR 描述和 commit message**：更浓缩，通常只有结论
4. **文档**：通常过时，没有人维护
5. **agent traces**：最完整，记录了完整的推理路径

每一层都比上一层更诚实，但每一层的维护成本也更高。agent traces 解决了信息丢失的问题，却带来了新的问题：这些海量数据该怎么索引？该怎么检索？哪个 trace 记录了真正重要的决策？

Hugging Face 的 buckets 解决的是存储问题，不是这个问题。

## 一个更根本的悖论

这整件事最有趣的地方在于：**我们正在用 AI 解决信息丢失问题，但 AI 本身也是信息丢失的原因之一**。

当人类工程师写代码，决策过程至少有部分发生在对话、文档、有意识的记录中。当 agent 写代码，决策过程发生在 token 序列里——这些序列被记录为 traces，但如果没有人主动查看和使用它们，它们就只是躺在存储里的字节。

记忆的前提是有人记得去回头看。traces 再完整，如果下一个工程师或 agent 从不看它，它就不存在。

也许真正的解决方案不是更好的存储，而是**更小的决策空间**——让每次 agent 调用解决的问题足够小、足够清晰，以至于不需要回顾太多历史就能做出好判断。

这是另一个形式的"本质复杂度 vs 偶发复杂度"。存储问题是偶发的；判断力问题才是本质。
