## 从"提示"到"循环"：一个范式转移

### 提示词的本质缺陷

当你给 Agent 一个提示词，你在做的是**单次输入**。无论提示词写得多详细，它本质上是一个"快照"——你在这个时间点，把你当前对任务的理解编码进去，然后期待 Agent 一次输出就满足你的所有要求。

这在逻辑上有一个根本问题：**很多反馈信息在任务开始时根本不存在。**

> 💡 **Key Insight**
>
> 很多反馈信息在任务开始时根本不存在——这不是模型的缺陷，而是任务的内在结构决定的。

比如 Bug，你只有在测试的时候才能知道它的存在。比如产品方向，你在看到初稿后才发现它遗漏了某个重要条件。这些信息不可能提前预知，因为它们是任务的内在组成部分，只有在任务执行过程中才会显现。

### 循环的不同

Agent Loop 的本质区别在于：它执行的不是"指令"，而是"目标"。

这个循环的每一步都不是固定的。Agent 需要观察当前状态，判断应该采取什么行动，执行行动后再观察结果，评估是否达到了预期，然后决定下一步怎么走。

<object data="/assets/images/2026-06-11-loop-engineering-02-five-components.svg" type="image/svg+xml" width="100%"></object>

> 💡 **Key Insight**
>
> 循环的每一步都是"观察—判断—行动—评估"的迭代，不需要提前覆盖所有情况。

这意味着：**你不需要预见所有情况，你只需要定义目标、评估标准和停止条件。**

## 工程师的位置变了

### 以前：工程师是操作者

在 Harness Engineering 阶段，工程师是系统的驱动者。每一轮任务都需要人工触发，人工判断对错，人工决定下一步。Agent 是工具，人是操作工具的手。

这带来的问题是：人的时间被低价值的重复劳动消耗。盯着 Agent 的每一次输出，判断对不对，想下一步怎么引导——这些事情技术含量低，但极度耗费注意力。

### 现在：工程师是设计师

Loop Engineering 的核心转变是：**工程师不再是循环的一部分，而是循环的设计者。**

你不需要在循环里，你站在循环外面设计它：

- 设计目标——什么算"完成"
- 设计上下文管理——每一轮给 Agent 什么信息
- 设计工具集——Agent 能调用什么
- 设计评估标准——怎么判断当前状态是否满足要求
- 设计停止条件——什么时候该停

这就像从"自己开车"变成"设计交通系统"。你不再亲自操控每一个转弯，但你需要设计好规则，让车辆能够自主运行。

> 💡 **Key Insight**
>
> 工程师不再是循环的一部分，而是循环的设计者——这是角色本质的转变，不是程度的转变。

## 五组件详解

### 目标（Goal）

目标定义是大多数 Loop 失败的原因。"帮我优化一下"不是目标，是请求。

一个好的目标应该是可验证的：

> "把这个接口的响应时间从 800ms 降到 300ms 以下。保留现有行为，所有测试必须通过。输出改动说明，列出具体做了哪些优化。"

这个目标的每一部分都是可验证的。清晰的目标给 Agent 提供了一个稳定的锚点，每一轮迭代都可以用这个锚点来校准。

### 上下文管理（Context Management）

上下文不只是对话历史。代码库的当前状态、相关文档、错误日志、测试结果、历史决策——这些都是上下文。

很多 Agent 表现差，根本原因不是模型不够聪明，而是上下文太脏（混杂无关信息）、太少（关键信息缺失）、或太随机（每一轮组织方式不一致）。

Ralph Loop 的重要创新：每次迭代都重置上下文到一组固定的锚点文件，而不是让对话历史无限增长。这解决了上下文污染的问题。

2026 年的 Loop 系统开始使用基于 git 的状态管理——每一轮的改动都会提交到 git，Agent 可以查看历史提交，理解之前做了什么、为什么要这么做。

### 工具（Tools）

工具的选择需要和任务匹配。给 Agent 写代码的能力但不给它运行测试的工具，它就无法验证代码是否正确。

但工具也不是越多越好。每增加一个工具，Agent 的决策空间就变大，它需要在更多选项中做选择。好的 Loop 设计会精心选择工具集，只提供完成任务必需的工具，每个工具都有清晰的用途和使用时机。

### 评估（Evaluation）

这是 Loop 的灵魂。没有评估，循环就会变成瞎转。

> 💡 **Key Insight**
>
> 没有评估，循环就会变成瞎转——评估让"完成"从声明变成可验证的证据。

评估的关键是自动化——如果每一轮都需要人来判断对错，Loop 就失去了自主运行的能力。

但自动化评估有局限。有些质量标准很难量化，比如代码的可读性、文字的流畅度。对于这些方面，需要引入人工检查点，让人在关键节点介入。

这里的核心洞察是：**好的 Loop 不是把人踢出去，而是把人放在最关键的检查点上。**

### 停止条件（Stop Conditions）

任何循环都需要退出条件。Agent 常犯的错误包括：

- **死循环**：反复尝试同样的方案，每次都失败，但不知道应该放弃
- **无限微调**：持续做微小的改动，每次都有一点点改进，但永远达不到完美，不知道应该停在哪里

所以需要多种停止条件：成功条件（所有评估都通过）、失败条件（连续多轮没有改进）、资源限制（运行时间超过上限、成本超过预算）、风险检查点（高风险操作前停下来等待人工确认）。

## 历史先例：从汇编到高级语言

这个转变有一个鲜为人知的历史先例。

<object data="/assets/images/2026-06-11-loop-engineering-03-assembly-to-loop.svg" type="image/svg+xml" width="100%"></object>

在计算机早期，程序员直接写机器码——每一条指令都是具体的操作，你想让计算机做什么，你就得一步一步告诉它做什么。后来有了汇编语言，稍微抽象了一点，但仍然是一一对应的关系。

然后出现了高级语言——C、Python、Java。程序员不再关心寄存器怎么用、内存怎么分配，而是关心"我想做什么"。编译器负责把这些高级意图翻译成机器能理解的指令。

Loop Engineering 是同一类转变的另一层：不是从"低级指令"到"高级指令"，而是从"单次指令"到"持续过程"。

提示词是汇编语言——你给出指令，Agent 执行，你再给出下一条指令。循环是高级语言——你定义目标，系统自主迭代直到达成目标。

## 无人值守的代价

Loop Engineering 有一个必须面对的风险：**无人值守状态下，AI 生成的代码可能出现质量问题。**

Token 成本的失控、输出质量的退化、认知负债的累积——这些都是实践者需要面对的真实问题。

最危险的情况是：你在设计 Loop，但你逐渐失去了对代码的理解。当你需要介入处理复杂故障时，你发现你已经无法判断系统行为的原因。

这意味着：**保持对代码的深刻理解，是设计者最重要的责任。**

## 谁来监督监督者？

斯坦伯格提出的 Loop，已经不单单是"让一个 Agent 反复做某个任务"，而是把 Loop 当成一种可以长期运行、互相协作、自动调度的 AI 工作系统。

当有多个 Loop 在同时运行时，新的问题出现了：**谁来协调它们？谁来决定优先级？谁来检查它们的工作质量？**

于是出现了用 Loop 去监督其他 Loop 的设计——通过一个总 Loop 负责观察全局，分发给多个子 Loop，总 Loop 检查它们的进度和结果。

这引出了一个更深的哲学问题：**当系统自主运行到一定程度，它的监督者还能理解它吗？**

## 结尾

Boris Cherny 删除 IDE 的那一刻，或许将被视为一个标志性的时刻。

但这个时刻的意义，不在于"AI 接管了代码编写"，而在于**人的位置发生了根本性的改变**。

从操作者到设计者，从输入者到架构者，从执行者到监督者。

这个转变的深层逻辑是：当你不再需要亲自操作，你真正需要的技能是什么？

> 💡 **Key Insight**
>
> 当你不再需要亲自操作，你真正需要的技能是判断力——知道什么是对的结果，知道什么时候该停，知道什么时候该介入。

这些技能，在任何时代，都是最稀缺的。
