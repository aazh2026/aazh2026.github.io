## Goal Mode 的本质

细心的人发现 Codex 的 `/goal` 命令已经可以用了。在提示词前加 `/goal`，Codex 会循环执行直到达成目标。

但真正用过的人知道：这个功能很容易失效。要么跑几分钟就停了，要么跑起来停不下来，漫无目的地乱改。

问题不在工具。在目标定义。

Goal mode 是一个循环：**Agent 执行动作、打分、检查分数是否满足目标、继续或终止。**

> 💡 **Key Insight**
>
> 关键在第三步：如果目标模糊，循环的终止状态就未定义。你怎么让 Agent 知道"让代码更好"什么时候算完成？什么状态算"够好"？

模糊目标有两种失败模式：

- **提前放弃**：模型跑了几分钟，感觉差不多了，停止
- **永不停止**：模型永远不停，漫无目的地持续修改

两种失败都是同一个原因：**目标没有清晰的终止状态。**

---

## 法则一：目标要量化

"让代码更好"是一个模糊目标。更好的目标是：

> "把 `specific_file` 里代码的运行时间降低 20%，且不引发任何现有单元测试和集成测试的回归。"

这就是一个清晰的量化目标。Agent 知道什么时候完成了。

如果你的目标本质上是定性的，转化成定量的形式。比如把"让论文格式正确"变成"执行一份 200 条格式规则的检查清单，200/200 全打勾才结束"。即使每条规则本身是模糊的，Agent 也能清楚地判断进度。

---

## 法则二：反馈循环要快

Agent 评估动作和目标之间的差距，需要一个测试机制。**测试跑得越快，模型得到反馈越快，迭代越有效。**

有一个真实的例子：有人做蛋白质结构模型架构搜索，原来跑一次评分要好几天。用了一个更小但采样良好的数据集替代完整训练集，把评分时间从几天压到几分钟。

> 💡 **Key Insight**
>
> 反馈循环的速度直接决定了迭代效率。瓶颈从来不是准确性，是迭代速度。

任何能加速反馈的方式都值得尝试：用更小的模型、用抽样数据、用更快的测试框架。前提是不损害评分质量。

---

## 法则三：给 Agent 文件来思考

Codex 可以连续跑几天。即使有内置的上下文压缩能力，模型在长时间跨度里保持连贯思路仍然很难。

与其强迫模型把所有上下文都记在脑子里，不如给它暴露 markdown 文件让它写。三个文件：

**PLAN.md** — 捕捉 Agent 在朝向目标移动时的高层计划。你可以用初始想法来播种它。

**EXPERIMENTS.md** — Agent 追踪每次实验细节的地方：标题、描述、结果。这是三个文件里最重要的。它让你和 Agent 都能回顾之前的尝试，知道什么有效、什么无效。

**EXPERIMENT_NOTES.md** — Agent 的 scratchpad。按时间顺序记录执行时的实时想法。这个文件让你能审查 Agent 的思维过程，在需要时把它拉回正确的方向。

---

## 完整玩法

设置一个清晰的可测量的目标。保持反馈循环紧。给 Agent markdown 文件来思考。

这三块到位了，Codex 可以为你的最难问题跑上几个小时，甚至几天。

> 💡 **Key Insight**
>
> 瓶颈从来不是 AI。是看你有没有给循环一个清晰的终止状态、一个快的反馈机制、和一个跨会话持久化思考的方式。你设计目标的方式，才是真正的门槛。
