# AI 编程代理的最后一公里：为什么"能跑"不等于"能用"

代码能跑。这是一个 AI 编程代理可以自豪地宣布的成果。但这句话的另一面很少被说出来：**能跑的代码和能上生产的代码之间，隔着一整套工程纪律。**

Addy Osmani 正在试图弥合这个鸿沟。他是 Google 工程总监，以往的声誉建立在前端性能优化和 JavaScript 设计模式上。他开源的 [agent-skills](https://github.com/addyosmani/agent-skills)（GitHub, MIT license）是一套给 AI 编程代理使用的工程技能库——24 个技能，覆盖从「确定做什么」到「生产上线」的全流程。

## 最短路径是默认设置

AI 编程代理有一个结构性问题：它们被训练成走最短路径。

给定一个任务，模型的激励机制很清晰——生成看起来能工作的代码，通过测试，然后继续下一个任务。这是合理的 RLHF 目标，但它的副作用被低估了：**工程师在写生产代码时，几乎每一步都在对抗这个激励。**

写代码之前先写 Spec？最短路径是直接开始写。写测试？最短路径是"反正我测过了"。安全审计？最短路径是"这应该没问题"。代码审查？最短路径是"能跑就行，不需要 review"。

这不是 AI 的缺陷。这是它的本质。

## Osmani 的解法：把工程纪律编码成工作流

agent-skills 的核心不是告诉 AI"要小心点"，而是把 senior engineer 的工作流固化成可执行的步骤。

8 条 slash commands 对应开发生命周期的关键节点：

```
Idea → Spec → Code → Test → QA → Go Live
/spec    /plan   /build  /test  /review  /ship
```

/spec 是 Spec-driven development——写代码之前先写 PRD，覆盖目标、接口、结构、代码风格、测试策略和边界。/build 是增量实现——每次只实现一个原子任务，测试驱动，验证通过后再继续。/review 是代码审查——五轴审查，变更大小控制在 ~100 行，Nit/Optional/FYI 严重级别标注。

24 个技能各有权衡（trade-offs）和反借口（anti-rationalization）表格。比如 TDD 技能的 rationalization 表里有一条：**"我会后面再加测试"**——对应的 rebuttal 是：**"没有测试的代码是负债，不是资产；积累负债的代码迟早要还。"**

这是关键设计。每个技能不只是"做什么"，还内建了"人类/AI 在这一步会找什么借口来跳过它，以及为什么那个借口是错的"。

## 历史先例：软件工程本身就是对"能跑就行"的反动

这个矛盾不是新问题。

1960 年代，软件的"软件危机"（Software Crisis）被提出来——代码规模超过人类管理能力，"它能跑"已经不够了。1972 年，Wirth 提出了 Wirth's Law：**软件变慢的速度比硬件变快的速度快。** 这个问题在 AI 时代变得更加尖锐：AI 生成代码的能力远超人类审查它的能力。

软件工程作为一个学科，就是在这种压力下被逼出来的。代码审查制度（code review）发明出来，是因为人们发现"合并后再修"比"提交前修"的成本高 10 倍以上。持续集成（CI）发明出来，是因为人工集成是瓶颈。测试金字塔（80% 单元测试 / 15% 集成测试 / 5% 端到端测试）被确立下来，是因为"全靠 E2E 测试"的团队都在生产事故里学会了教训。

每一种工程实践，都是因为有人走了最短路径，然后付了代价，才被发明出来。

## 深层机制：最短路径在复杂系统里是局部最优、全局最差

为什么最短路径在软件工程里总是出问题？

因为软件系统是复杂适应系统（complex adaptive systems）。每个局部最优的选择，累积起来经常导致全局最差的结果：模块之间的高耦合让改一个 bug 引入三个；没有测试的代码在重构时寸步难行；没有安全审计的接口暴露了生产数据；没有性能基准的优化是纯粹的自我感动。

AI 编程代理擅长找局部最优——当前上下文里看起来对的解。但生产软件的质量是在时间维度上定义的：一段代码今天能跑，三个月后还能跑吗？别人接手时能理解吗？攻击者能找到入口吗？流量翻倍时能撑住吗？

这些问题的答案不能从"当前任务的最短路径"里涌现出来。它们需要刻意绕远路——写 Spec 是绕远路，写测试是绕远路，做 code review 是绕远路。但这些远路在复杂系统里的回报是正的，而且随时间增长。

## 结尾

Osmani 的 agent-skills 给 AI 装上了一套"工程直觉"。它不是让 AI 变得更聪明，而是让它变得更像一个有纪律的 senior engineer——知道什么时候不能走最短路径。

这个项目的真正价值不是 24 个技能本身，而是它揭示的那个结构性矛盾：**AI 编程代理的能力上限取决于它对抗自身最短路径本能的程度。** 技能越多，对抗越系统化；对抗越系统化，代理产出的代码离生产级质量越近。

这大概是软件工程史上第二次，有人需要把"工程纪律"这个东西显式地教给工具。第一次是教人类；这一次是教机器。
